# Параметры конфигурации

Конфигурация — это точная настройка поведения JVM под конкретное железо. Автоматически сгенерированный `default.json` покрывает ~95% случаев: утилита смотрит на ваш CPU, количество ядер, объём RAM, размер L3-кэша и подставляет значения которые хорошо себя показали на живом STALCRAFT.

Этот документ объясняет **зачем** существует каждый параметр и **в какую сторону** его крутить, если хочется докрутить вручную. Конкретные числа смотрите в `configs/default.json` после первого запуска — там уже стоят дефолты под ваше железо.

> **Предупреждение:** неправильно выставленный параметр часто даёт результат **хуже** чем "не трогать вообще". Без чётко сформулированной проблемы оставьте автосгенерированный конфиг. Каждая ручная правка должна быть осознанной.

## Память

### `heap_size_gb`

Фиксирует размер heap'а JVM (`-Xmx`) в гигабайтах. Heap — это место, где живут все Java-объекты: чанки, сущности, буферы текстур, структуры мира. Чем больше heap, тем реже срабатывает GC, но тем дольше каждая пауза (G1 обходит весь heap). Живой working set STALCRAFT редко превышает 4 GB, поэтому бесконечно наращивать heap вредно.

Утилита выбирает от 2 GB (на 8 GB total RAM) до 8 GB (на 24+ GB total RAM). **Руками трогать почти никогда не нужно.** Если у вас 16 GB и STALCRAFT сообщает `OutOfMemoryError` (очень редко, только при агрессивных модах) — можно попробовать 6 или 7. Больше 8 ставить не стоит: это увеличит длительность mixed GC, что сломает плавность фреймтайма.

### `pre_touch`

Заставляет JVM при старте процесса физически закоммитить все страницы heap'а вместо ленивой аллокации. Без этого флага Windows выдаёт страницы по мере обращения к ним, и первый доступ к каждой странице во время игры вызывает page fault → микрофриз на 1–5 мс.

**Включайте (`true`)** если у вас **12+ GB RAM**. На слабых системах PreTouch съест слишком много RAM в момент запуска, что вредит больше чем помогает (Windows начнёт свопить). Минус — игра стартует на 1-3 секунды дольше, потому что все 4-8 GB touch'атся сразу.

### `metaspace_mb`

Размер области для метаданных загруженных классов. STALCRAFT держит в памяти ~11 тысяч классов (движок + ресурспаки + lambda-generated classes + reflection accessors), каждый занимает 10-15 KB метаданных. Суммарно 150–220 MB пиково.

**Ставьте 512 MB.** Меньше — риск `OutOfMemoryError: Metaspace` при загрузке больших ресурспаков или после долгой сессии (где reflection нагенерировал много hidden классов). Больше — бесполезно, пустая резервация. Утилита фиксирует `MetaspaceSize = MaxMetaspaceSize`, чтобы JVM не делала периодические расширения (каждое расширение = full GC = катастрофа).

## G1 GC — базовые параметры

### `max_gc_pause_millis`

Целевое значение максимальной длительности GC паузы, в которое G1 пытается уложиться. G1 динамически подстраивает размер young gen и количество регионов в mixed GC так, чтобы паузы не превышали этот таргет.

**Меньше — плавнее фреймтайм, но больше GC.** На обычном железе 35 мс даёт хороший компромисс: пауза на один кадр при 60 FPS (16.7 мс frame time — два кадра подряд), игрок это почти не замечает. На мощных CPU (X3D-классы с 96 MB L3) можно опускать до 20-25 мс, получая полностью плавную игру. На слабых CPU не опускайте ниже 30 мс — G1 просто не успеет уложиться, и начнёт делать паузы всё равно, но под нагрузкой throughput упадёт.

### `g1_heap_region_size_mb`

Размер одного региона G1. Heap делится на равные регионы, из которых состоит вся механика G1. Допустимые значения — степени двойки от 1 до 32 MB. Меньшие регионы дают G1 более тонкий контроль над mixed GC (можно выбрать точнее, какие регионы чистить), большие регионы экономят на RSet-сканировании.

**Утилита выбирает размер от heap'а** (4 MB для heap 2-3 GB, 8 MB для 4-5 GB, 16 MB для 6-7 GB, 32 MB для 8 GB). Руками трогайте только если хотите более гранулярный mixed GC для low-latency (8 kHz мышь, e-sport) — тогда ставьте на один шаг меньше.

### `g1_new_size_percent` и `g1_max_new_size_percent`

Минимальная и максимальная доли heap'а, которые могут быть отданы под young generation (Eden + Survivor). Young gen — это место где создаются все новые объекты, большая часть умирает молодой, маленькое количество промотируется в old gen.

Больший young gen = меньше частых young GC, но дольше каждый. **Ставьте 30% / 50%** — утилита так и делает. Эти значения оптимальны для STALCRAFT с его высоким allocation rate. Уменьшение ниже 20% / 40% даст частые минорные паузы, увеличение выше 40% / 60% съест heap'овый запас под old gen.

### `g1_reserve_percent`

Доля heap, которую G1 резервирует "на чёрный день" — под peak-аллокации между GC циклами. Если эта резервация кончится, сработает аварийный Full GC, который останавливает игру на сотни миллисекунд.

**Ставьте 20%.** Меньше — риск full GC на пиках аллокаций (эффекты, партиклы, массовая загрузка чанков). Больше — неэффективное использование heap'а. STALCRAFT с его неровным allocation rate любит запас.

### `g1_heap_waste_percent`

Порог терпимости G1 к "мёртвому" пространству в heap. Если в старых регионах больше чем X% мусора, G1 начинает активнее их собирать через mixed GC. Меньшие значения делают G1 агрессивнее.

**5% — хороший дефолт.** Агрессивная уборка предотвращает накопление мусора, но не настолько агрессивная, чтобы mixed GC стал постоянным. Можно попробовать 10% если mixed GC паузы мешают, но ценой — чуть больше памяти будет "впустую" занято мусором.

### `g1_mixed_gc_count_target`

На сколько циклов mixed GC размазать сбор старой генерации. G1 распределяет работу по N последовательным циклам, чтобы каждый был короче.

**4 — стандарт для throughput, 8 — для низкой задержки.** Больше циклов = меньше каждая пауза, но больше общее количество пауз. Для STALCRAFT с упором на плавный фреймтайм 8 — отличный выбор на мощных CPU. На слабых CPU 4 даст более стабильный throughput.

### `initiating_heap_occupancy_percent`

Порог заполнения heap, при котором G1 стартует concurrent marking cycle — фоновое сканирование old gen для подготовки к mixed GC. Слишком высокий порог = concurrent marking не успеет, сработает full GC. Слишком низкий = постоянное фоновое CPU usage.

**20% — безопасный дефолт, 15% — для мощных систем.** На X3D-классах с большим L3 и быстрой памятью можно позволить себе более ранний старт — concurrent marking сделается быстрее, и mixed GC успеет до того как heap заполнится. На слабом железе 20-25% даст больше времени на фактический concurrent phase.

### `g1_mixed_gc_live_threshold_percent`

G1 включает старый регион в mixed GC только если в нём меньше чем X% живых объектов (остальное — мусор, есть что чистить). Смысл — не тратить усилия на почти-полные регионы, где выгода минимальна.

**90 — правильное значение.** Низкие значения (65-85) из старых гайдов оставляют много мусора в heap'е. 90% даёт G1 свободу чистить даже плотные регионы, если в них всё-таки есть место для освобождения — это предотвращает накопление long-lived объектов.

### `g1_rset_updating_pause_time_percent`

Какой процент GC паузы G1 разрешено тратить на обновление Remembered Sets (структура, которая трекает cross-region references). Меньше — короче пауза, но часть работы переносится в concurrent phase (фоновая нагрузка на CPU).

**0% — всё concurrent.** На 6+ ядрах concurrent refinement thread'ы справляются сами, не нужно делать это в STW. На 4-ядерных можно выставить 3-5%, чтобы разгрузить фон.

### `survivor_ratio`

Соотношение между Eden и каждой Survivor областью внутри young gen. При `32` получается: Eden = 32 × Survivor, то есть Survivor очень маленький. Это значит что объекты быстрее промотируются в old gen (через Survivor они не задерживаются).

**32 — наша философия "быстрая промоция".** Классические гайды рекомендуют 6-8 (большие Survivor = объекты долго живут в young), но для STALCRAFT с его коротким жизненным циклом временных объектов быстрая промоция выгоднее: меньше копирования между Survivor-областями, меньше bookkeeping.

### `max_tenuring_threshold`

Максимальное количество young GC, которое объект должен пережить прежде чем его переведут в old gen. С `1` любой объект выживший первый young GC сразу идёт в old.

**1 — парный параметр к `survivor_ratio: 32`.** Вместе они реализуют стратегию "объект либо умер очень быстро, либо его сразу в old gen". Это идеально для STALCRAFT: большинство временных объектов (точки партиклов, bounding boxes, временные коллекции) умирают в Eden, а те что выжили один цикл — скорее всего долгоживущие (entities, chunks) и их нет смысла держать в young.

## G1 GC — продвинутые флаги минимизации STW

Все параметры ниже **experimental** — требуют `-XX:+UnlockExperimentalVMOptions`, который утилита подставляет автоматически.

### `g1_satb_buffer_enqueuing_threshold_percent`

Порог заполнения SATB (Snapshot-At-The-Beginning) буфера, при котором G1 начинает активно его обрабатывать. SATB — это способ, которым G1 отслеживает изменения графа объектов во время concurrent marking.

**30 — разумно.** Обработка раньше = меньше накопления работы, меньше длинных пиковых пауз. 0 отключает оптимизацию полностью.

### `g1_conc_rs_hot_card_limit`

G1 маркирует карточки памяти с частыми updates как "горячие" и держит их отдельно. Этот параметр — порог когда карточка становится горячей. Горячие карточки обрабатываются чаще и не попадают в общую очередь refinement.

**16 — дефолт.** Работает хорошо в большинстве случаев. Повышать стоит только если видите что `g1_conc_refinement_service_interval_millis` тратит много CPU.

### `g1_conc_refinement_service_interval_millis`

Интервал между циклами фоновой обработки hot cards. Меньше = G1 отзывчивее, больше = меньше фоновой CPU нагрузки.

**150 мс — баланс.** Игра не чувствует задержки от этого интервала, CPU overhead минимален. Не трогайте.

### `gc_time_ratio`

Целевое отношение времени работы приложения ко времени GC. Значение 99 означает "на 99 минут работы игры допустимо 1 минута GC" = 1% overhead. Это используется G1 для адаптивных решений.

**99 — стандарт.** Трогать не нужно, просто оставьте.

### `use_dynamic_number_of_gc_threads`

Позволяет G1 автоматически менять количество активных GC-воркеров в зависимости от нагрузки. Полезно на современных CPU с P+E ядрами (Intel 12+), где G1 может мигрировать между ядрами разной производительности.

**Включайте (`true`)** везде кроме самых слабых CPU. На 4-ядерных экономия не стоит вариативности задержек.

### `use_string_deduplication`

G1 фоном находит идентичные `String` объекты и объединяет их внутренний массив char'ов в один shared. STALCRAFT создаёт тонны одинаковых строк (tag names, translation keys, item IDs), дедупликация экономит 100-200 MB heap'а за длинную сессию.

**Включайте (`true`)** если у вас 8+ ядер. На слабых CPU ~1% CPU overhead от dedup thread может быть заметен, но на мощных это чистый выигрыш heap.

## Потоки GC

### `parallel_gc_threads`

Количество воркеров для STW фаз (young GC, mixed GC copy phase). Эти потоки активны только во время паузы, поэтому их количество прямо влияет на длительность пауз: больше потоков = параллельнее работа = короче пауза.

**Правило: `физические ядра - 2`, но не больше 10 и не меньше 2.** Оставляем 2 ядра главному game thread и render thread. Верхний предел 10 — у G1 diminishing returns после этого на consumer CPU. Утилита считает это автоматически.

### `conc_gc_threads`

Количество воркеров для фоновых (concurrent) фаз: concurrent marking, concurrent refinement, SATB processing. Эти потоки работают параллельно с игрой, воруя у неё CPU.

**Обычно `parallel / 4`, минимум 1, максимум 4.** Больше concurrent воркеров = быстрее concurrent phase = меньше риск full GC, но забирают ядра у игры. На X3D-классах утилита добавляет +1 к дефолту — мощные ядра могут позволить себе концуррентную работу без ущерба FPS.

### `soft_ref_lru_policy_ms_per_mb`

Соотношение "сколько времени в миллисекундах JVM держит soft references в памяти на каждый MB свободного места в heap". Влияет на то, как быстро JVM чистит soft-reference кэши (например, кэш текстур LWJGL).

**50 — разумно.** Больше = кэши живут дольше, меньше GC на их восстановление, но heap занят дольше. Меньше = агрессивная очистка, больше повторной работы. Значение по умолчанию в JVM — 1000, оно избыточно для игр с ограниченным heap.

## JIT компиляция

JIT (Just-In-Time) компилятор JVM на лету превращает bytecode в нативный машинный код. В OpenJDK это C2 — агрессивный optimizing compiler. Его настройки определяют, насколько агрессивно JVM будет оптимизировать ваш код. Агрессивнее = плавнее игра, но больше памяти под code cache и дольше прогрев.

### `reserved_code_cache_size_mb`

Максимальный размер кэша для скомпилированного JIT-кода. Когда кэш заполняется, JIT перестаёт компилировать новые методы и начинает выбрасывать старые — это катастрофа для стабильности FPS.

**400 MB — безопасный запас.** STALCRAFT реально использует 150-250 MB из этого, остальное — headroom на случай если reflection нагенерировал много скомпилированных accessor'ов. Меньше 256 MB ставить не стоит — будете упираться в потолок.

### `max_inline_level`

Глубина вложенного inlining'а — сколько уровней вызовов C2 может "раскрыть" прямо в код вызывающего метода. Более глубокий inlining = быстрее hot path, но больше размер скомпилированного кода.

**15 для обычных CPU, 20 для X3D-классов с большим L3.** С 96 MB L3 кэша можно позволить себе агрессивный inlining — весь hot path всё равно влезет в кэш. На обычных CPU с 16-32 MB L3 глубокий inline вытеснит hot data из кэша, и выиграешь в одном месте, проиграешь в другом.

### `freq_inline_size`

Порог размера "горячего" метода, который JIT разрешено инлайнить несмотря на размер. Обычные методы имеют более строгий лимит размера для inlining, но часто вызываемые методы получают эту увеличенную квоту.

**500 для обычных CPU, 750 для big-cache.** Парный параметр к `max_inline_level` — оба определяют, сколько кода уйдёт в inlined-версии hot path.

### `inline_small_code`

Порог размера скомпилированного метода, ниже которого JIT считает его "маленьким" и агрессивно инлайнит. Большое значение = больше методов попадает под аггрессивный inline.

**4000 для обычных, 6000 для big-cache.** Тот же паттерн — больше кэша CPU = можно позволить себе больше скомпилированного кода в inline.

### `max_node_limit` и `node_limit_fudge_factor`

`max_node_limit` ограничивает максимальное количество узлов в IR-графе C2 для одного метода. Сложные методы (render loop, chunk mesher) могут упереться в этот лимит и остаться некомпилированными — это означает выполнение в интерпретаторе, т.е. ~10-100x медленнее. `node_limit_fudge_factor` — это допуск сверх лимита, который C2 может взять при оптимизации.

**240000 / 8000 для обычных CPU, 320000 / 8000 для big-cache.** Эти значения позволяют C2 компилировать даже самые тяжёлые методы STALCRAFT. Меньшие значения (дефолт JVM — 80000) оставят несколько важных методов интерпретированными.

### `nmethod_sweep_activity`

Интенсивность очистки code cache от устаревших методов. Значение 1 — минимальная очистка, 4 — агрессивная.

**1.** Методы STALCRAFT после прогрева не устаревают — один раз скомпилированы и живут до выхода. Агрессивная очистка только вызывает ненужные повторные компиляции.

### `dont_compile_huge_methods`

Запрещает JIT-компиляцию методов превышающих внутренний порог "огромности" (~8000 bytecode инструкций).

**`false`.** STALCRAFT имеет несколько огромных методов (chunk renderer, entity AI), и их критически важно скомпилировать. `true` = эти методы останутся в интерпретаторе = постоянные просадки FPS в соответствующих сценах.

### `allocate_prefetch_style`

Стратегия software prefetch при аллокации новых объектов. C2 генерирует `prefetch` инструкции перед TLAB-аллокациями, чтобы заранее подтянуть память в cache line.

**3 — максимально агрессивно.** На современных CPU цена prefetch инструкции околонулевая, а эффект на allocation-heavy workload заметен. Значение 0 отключает prefetch совсем — не нужно.

### `always_act_as_server_class`

Заставляет JVM всегда использовать server JIT (C2) вместо client JIT (C1) для top-tier компиляции. На Windows JVM определяет "серверность" автоматически по железу, и иногда ошибается.

**`true`.** Гарантирует что C2 включится даже на нетипичных конфигурациях. Увеличивает время прогрева, но для долгой игровой сессии это однократная цена.

### `use_xmm_for_array_copy`

Использовать XMM (SSE2) регистры для копирования массивов. Это SIMD-инструкции, которые копируют 16 байт за такт вместо 8.

**`true`** на всех CPU новее Pentium 4. Чистый выигрыш для любой копирующей операции (String.clone, Array.copy, мемори-операции в buffer'ах LWJGL).

### `use_fpu_for_spilling`

Разрешает C2 использовать FPU/SSE регистры для spill'а значений когда регистры общего назначения заканчиваются. Это альтернатива сохранению значений на стеке.

**`true`.** Spill в FPU быстрее чем на стек (нет обращения к памяти), это стабилизирует фреймтайм в сценах с большой нагрузкой на регистры CPU.

## Java 9 специфика

Эти параметры специфичны для OpenJDK 9 (версия которую использует STALCRAFT). В более поздних версиях Java они могут работать по-другому или вообще отсутствовать.

### `reflection_inflation_threshold`

По умолчанию JVM для первых 15 вызовов reflection-метода использует медленный интерпретируемый путь, и только после этого генерирует быстрый bytecode accessor. Это "прогрев" reflection, и он стоит времени на старте игры.

**0 = сразу компилировать.** STALCRAFT активно использует reflection в event bus, config loading, mixin loader — прогрев занимает ощутимое время. Установка в 0 эти ~15 вызовов на метод экономит, старт игры становится заметно быстрее.

### `auto_box_cache_max`

JVM кэширует объекты `Integer.valueOf(n)` для диапазона [-128, 127] — это называется autobox cache. STALCRAFT использует `HashMap<Integer, ...>` для block IDs, chunk coords, packet IDs — а эти числа часто вне стандартного диапазона.

**4096.** Расширяет кэш до [-128, 4095] — теперь все block IDs (их примерно 2000 в ванили + моды) попадают в кэш, и каждый `Integer.valueOf(blockId)` не создаёт новый объект. Это убирает миллионы аллокаций в hot path рендера и сетевого кода.

### `use_thread_priorities` и `thread_priority_policy`

Позволяют JVM транслировать `Thread.setPriority()` из Java в реальные приоритеты потоков Windows. По умолчанию Windows JVM зажимает все потоки в `NORMAL`, игнорируя вызовы setPriority.

**`true` + policy `1`** разблокируют полный диапазон приоритетов. LWJGL render thread и main game loop получают более высокий приоритет чем GC worker'ы, что даёт стабильнее фреймтайм. Policy 1 = "aggressive" — использует все уровни Windows приоритетов, включая выше NORMAL.

### `use_counter_decay`

JVM раз в ~10 секунд уменьшает метрики "горячности" JIT-методов (periodic counter decay). Идея в том, чтобы "старые горячие" методы могли уступить место "новым горячим" — но в игре все горячие методы (render, AI, physics) горячие всё время сессии.

**`false` = отключить decay.** Счётчики накапливаются монотонно, горячие методы остаются навсегда скомпилированными, никаких повторных компиляций из-за того что метрика "остыла".

### `compile_threshold_scaling`

Множитель порога C1→C2 промоции. Значение 1.0 — стандарт (~10000 вызовов метода до C2), 0.5 — в два раза раньше (~5000 вызовов).

**0.5 = быстрее прогрев.** Метод быстрее попадает в финальную C2-версию, игра быстрее достигает peak performance после загрузки. Минус — чуть больше CPU во время прогрева (первые минуты игры), но это однократный костер за более стабильный геймплей.

## Large Pages

### `use_large_pages`

Включает использование 2 MB (или 1 GB) страниц памяти вместо стандартных 4 KB. Большие страницы уменьшают TLB pressure — процессор меньше тратит такты на page table walks. Выигрыш для allocation-heavy приложений вроде STALCRAFT — 2-5% throughput и меньше TLB-jitter (критично для latency-sensitive сценариев типа 8 kHz мыши).

**Требует настройки Windows.** Без прав `SeLockMemoryPrivilege` JVM молча проигнорирует флаг:

1. `Win + R` → `gpedit.msc`
2. Computer Config → Windows Settings → Security Settings → Local Policies → User Rights Assignment → **Lock pages in memory**
3. Add User → ваша учётка
4. **Перезайти в Windows** (политика применяется на логоне)

После этого поставьте `use_large_pages: true`. Утилита сама проверяет наличие привилегии при генерации конфига — если её нет, параметр выставляется в `false` и не даёт ложных ожиданий.

На Windows Home (без `gpedit.msc`) настроить Large Pages штатными средствами нельзя — оставьте `false`.
